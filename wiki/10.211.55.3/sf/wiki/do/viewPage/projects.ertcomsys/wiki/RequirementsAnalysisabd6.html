wiki1016: RequirementsAnalysis (Version 25)
Requirements artf1220, artf1216 and artf1219 together form the extended StartOS() functionality.

Requirements artf1231, artf1232, artf1233 and artf1234 ("constructional elements") are relics of a pre-OIL OSEK spec (the intention was that the source code of a system was inspected by a tool to determine the elements of a system; this approach has been replaced by OIL). They no longer have any semantics and are present for backwards-compatibility only.

Requirement artf1045 appears to show a misunderstanding of how the OSEK OS works. Tasks can run at interrupt level by getting a resource shared with an ISR. Thus this requirement cannot be satisfied without breaking OSEK compatibility. Probably what the designers were getting at was the problem of making API calls bracketed by the fast interrupt functions (SuspendAllInterrupts etc.). This might be a better approach: to check that these calls are not outstanding when an API call is made. Note: this has been fixed in AUTOSAR 3.0 by a rewording of requirement OS093 to the interpretation just given here.

Requirement artf1280 is problematical for library (binary) components. The handle is embedded in the name of the call (obviously for speed) but to get these calls requires that the configuration process has run and the handles of the configured system are available. This means that it is not possible for a library function, given a message handle from the environment, to access the flags. It is probably appropriate to create a flag control block or find some other way of accessing flags, perhaps via the message object directly. Perhaps a new Flag API.

Requirement artf1100 needs RES_SCHEDULER to be created. For the sake of library code, RES_SCHEDULER should be defined in oslib.h rather than be created by the configuration process. This is because a library component can see nothing of the configured elements since it has no access to handles. Similarly, requirement artf1207 defines OSTICKDURATION. As with other requirements, this must be an externally-linked variable rather than a #define from the configuration tool: the library must patch these at link time since at compile time no handles are available. The same is true for artf1221: OSDEFAULTAPPMODE.

AUTOSAR 3.0

In AUTOSAR 3.0 the requirements artf1033, artf1037, and artf1047 have been deleted. New requirements are OS358 and above. There are 42 new AUTOSAR 3.0 Class 1 requirements (in addition to 59 requirements carried over from AUTOSAR 2.0 Class 1). There is a requirements matrix that shows the AUTOSAR 3.0 requirements.

Requirement OS427 (artf1377): The Operating System shall allow the Final Delay for a single-shot schedule table to be zero. This contradicts requirement OS444 (artf1383): The value of Final Delay shall be in the range OsCounterMinCycle .. OsCounterMaxAllowedValue of the underlying counter. (assuming OsCounterMinCycle is greater than zero). Normally this wouldn't be an issue because the Final Delay value only has meaning for periodic schedule tables. But a one-shot schedule table can be 'nexted' (i.e. another schedule table is kicked off at the end of the current one). This means that the subsequent one could overlap with the end of the current one, giving simultaneous expiry points, leading to non-determinism in which expiry point is processed first (something that OS412 (artf1371) is trying to overcome) and also defies the requirement that expiry points are at unique times, contradicting OS442 (artf1381). The result of this is that OS427 must be ignored.

The requirement OS276 (artf1054) prohibits a total delay of OsCounterMaxAllowedValue at run-time by returning E_OS_VALUE, which is correct behaviour but inconsistent with the constraints placed on Final Delay and Initial Offset, which do not have to sum to less than or equal to OsCounterMaxAllowedValue. See OS443 (artf1382) and OS444 (artf1383). In any case, these constraints would be difficult to enforce offline because they would relate to Final Delay of one schedule table and Initial Offset of a different one (see OS284 for details). It is possible to overcome this by implementing a system where a 'dummy' initial expiry point at zero is inserted which does nothing and merely serves to join two counter delays together, ensuring that Initial Offset in the implemented system is always zero and thus ensuring that the total delay never exceeds OsCounterMaxAllowedValue. See artf1387.

The requirement OS412 (artf1371) and the narrative before it says that expiry points must be processed as tasks-then-events to ensure determinism. This is not true, however, since the ordering of task activation is not defined. Since tasks are processed in a FIFO order at the same priority by OSEK then it is possible to have two different execution orders in two different implementations.

OS402 (artf1363) and OS403 (artf1364) use the noun "set" when discussing tasks and events (although note that events are tuples), and therefore it can be assumed that a task or event should appear at most once in an expiry point definition. The OIL specification for v2.0 did not impose this requirement, since expiry points were made up in OIL of a collection of actions with the same offset.

There is a 'caveat' in the specification relating to OS382 (artf1354). It says if the timer already passed the <Value> value a second (or multiple) time, the result returned is wrong.. This is slightly misleading, since passed is the wrong word to use: if the timer becomes equal to Value then the result returned is zero, and this is also wrong in the case of a full wrap of the timer (also, for timers of the full range of TickType it is impossible within the type system to return a value longer than the full range).

OS368 (artf1347) requires that the error hook be called (if configured) in the case where a cat2 ISR returns from its function without properly releasing interrupts (that have been locked by the 'fast' interrupt macros). This is a vague requirement and some interpretation is required. The requirement appears to operate in standard status as well as extended, even though this would slow down all interrupts in standard status (which is supposed to operate quickly). The purpose of standard status is that all the deterministic errors are removed by testing using extended status and these checks are not included in standard status. However, this philosophy does not appear to have held with other requirements (e.g. artf1042) so for consistency we shall assume that requirement OS368 operates in both standard and extended status. Another open issue is the service ID code to report in the error hook. Because there is no natural API call to associate this error with, a new service ID code is invented (OSServiceId_NoServiceId) and this is what will be reported in the error hook with the OSErrorGetServiceId() call. Requirement OS069 (artf1041) is updated to use this code.

OS369 (artf1348) is a Cat2 ISR version of OS070 (artf1042), but also includes the requirement to call the error hook if there are locked resources. In standard status in the Scalios implementation it is not possible to even detect this condition. Why the version is added, and then different to OS070, is not clear. Note also that this requirement, combined with OS368, massively extends the worst-case latencies of the ISRs.
