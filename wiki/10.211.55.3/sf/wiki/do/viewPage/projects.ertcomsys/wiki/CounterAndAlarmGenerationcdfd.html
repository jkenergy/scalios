wiki1042: CounterAndAlarmGeneration (Version 17)

Counter generation

A ROM array called os_counters[] is generated, with each element corresponding to a counter defined in OIL. Element 0 is the pre-defined system counter.

The ROM pointers os_first_counter and os_last_counter are generated if the build level is extended status (these are used in extended status checks):

const struct os_countercb *os_first_counter = &os_counter[0];
const struct os_countercb *os_last_counter = &os_counter[5];

The ROM integer os_num_counters is generated and equal to the size of the ROM os_counter[] array.

If the number of counters is zero then a dummy array is generated but os_first_counter and os_last_counter are set to 0. Note that the first counter defined is presumed to be the system counter.

Three externed variables need to be instantiated:

const TickType OSMAXALLOWEDVALUE = nnn;
const TickType OSTICKSPERBASE = nnn;
const TickType OSMINCYCLE = nnn;

Where nnn represents the corresponding values defined in OIL for the system counter.

In addition, macro definitions should be created in oshandles.h for all counters (including the system counter) of the following form:

#define OSMAXALLOWEDVALUE_x (nnnU);
#define OSTICKSPERBASE_x (nnnU);
#define OSMINCYCLE_x (nnnU);

where x represents the corresponding handle name of the counter as defined in OIL.

For each element of the ROM os_counter[] array a counter control block (struct os_countercb) is generated. This is generated two ways, depending on whether the counter is a singleton counter or a multi counter.

For singleton counters

The dyn_m field is set equal to 0.

The singleton_alarm field contains the handle for the singleton alarm bound to the counter.

The expired field points to the function os_counter_expired_single().

The setrelalarm field points to the function os_setrelalarm_single().

The cancel field points to the function os_cancel_alarm_single().

For multi counters

The dyn_m field is set to point to a corresponding os_countercb_multi_dyn RAM struct (for each multi counter). This structure is initialized to zero and so no initializers appear in the generated code. The configuration tool is free to generate this structure in a RAM array, or as individually-declared structures.

The singleton_alarm field contains 0.

The expired field points to the function os_counter_expired_multi().

The setrelalarm field points to the function os_setrelalarm_multi().

The cancel field points to the function os_cancel_alarm_multi().

For both singleton and multi counters

The alarmbase field points to a ROM os_alarmbasecb struct. This contains the alarmbase definition.

The stop, start, disable_ints, and enable_ints are function pointers that point to device-specific driver functions that control the underlying hardware of the counter. The field device is a pointer to a device_specific structure that contains details of the underlying hardware, used by the device driver functions. These are part of the counter device driver API.

For software counters

A RAM struct os_cdevice_softcb_dyn is created for each software counter. The RAM struct is initialized to zero (so no initializer code is generated). The device field of the ROM os_countercb struct points to this RAM structure. In the initialization of the ROM struct a cast of the pointer to an os_cdevicecb pointer is necessary.

For hardware counters

The generation of the structures is device-specific.

Alarm generation

A ROM array called os_alarms[] is generated, with one entry per alarm declared in OIL. If there are no alarms declared then the array must still be generated, but with a dummy entry (that can have any values).

A ROM value called os_num_alarms is generated which contains the number of alarms declared in OIL (this may be zero).

If the reinit() function is present then two ID check ROM pointers are generated. Example:

const struct os_alarmcb *os_first_alarm = os_alarms[0];
const struct os_alarmcb *os_last_alarm = os_alarms[4];

If there are no declared alarms (os_num_alarms is 0) then os_first_alarm and os_first_alarm are also 0.

Each entry in the os_alarms[] array is filled depending on whether the alarm is a singleton alarm or a multi alarm.

Singleton alarm generation

The m subfield of the dyn field is 0.

Multi alarm generation

The m subfield of the dyn field points to a os_alarmcb_multi_dyn struct. One of these structs is created for each multi alarm (they can be created individually or in an array; the generation tool is free to decide). The structs are initialized to zero and so no initializer code is generated.

Common fields to both singleton and multi alarm generation

A RAM os_alarmcb_common_dyn struct is generated for each alarm and initialized to zero (no initializer code is generated).

The c subfield of the dyn field points to the corresponding RAM os_alarmcb_common_dyn struct.

The counter field of each alarm control block contains the handle of the counter to which the alarm is bound.

The process field points to a function that processes the expiry of the alarm. Which function depends on the alarm expiry action configured in OIL, as follows:

  • os_expiry_activatetask() if the alarm activates a task.
  • os_expiry_setevent() if the alarm sets an event.
  • os_expiry_callback() if the alarm calls a callback function.
  • os_expiry_incrementcounter() if the alarm increments a software counter.
  • os_expiry_schedtab() if the alarm processes a schedule table.

The action field is a union, initialized depending on the alarm expiry action configured in OIL as follows:

  • If the alarm activates a task then the task subfield of action is set to the handle of the task to be activated.
  • If the alarm sets an event then the event subfield of action is set to the event mask and the handle of the task to which the event is set.
  • If the alarm calls a callback function then the callback subfield of action is set to point to the callback function (named os_alarmcallback_nnnn) where nnnn is the callback name configured in OIL.
  • If the alarm increments a software counter then the counter subfield of action is set to the handle of the software counter to be incremented.
  • If the alarm expiry processes a schedule table then the schedtable subfield of action is set to the handle of the schedule table to be processed.
