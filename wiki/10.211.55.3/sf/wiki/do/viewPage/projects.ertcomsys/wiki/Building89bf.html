wiki1078: Building (Version 7)

Embedded code building

The OS is build via make. A makefile called makeversion.mf is used to build a specific library (see library build levels), and a makefile called build.mf builds all four build levels of Scalios. The target-specific aspects of building are contained in the target.mf makefile in the target-specific folder and included into makeversion.mf by make itself.

There are also source files that are compiled several times to include optimized versions of functions into each library, or to provide several functions from virtually identical source code. These files are:

  • counters/setalarm.c. See shared alarm code for details.
  • kernel/queue.c. See task queueing optimization for details.
  • kernel/disp.c. See dispatcher optimization for details.
  • kernel/slih.c. See SLIH for details.
  • schedtables/startschedtab.c.

They are compiled by copying the source code to a different file and prepending a pre-processor #define. This is done instead of using #include to share source code because many embedded systems debuggers fail to debug code that has been #included.

There are specific build options for enabling assertion checks, etc. See library build levels for details.

target-specific builds: dspic. interrupt wrappers. in own hierarchy.

visibility: OS, applications, internals.

Host tool building

Eclipse-based. Autobuild, autocreate the JAR file. JAR file runs as a parameter to standard java. Futzing with versioning of JVM when running from within Eclipse: must use same VM as Eclipse.

Test building

All tests are built within a standard test framework that takes care of the build environment across all targets. They are designed to run quickly (taking no more than a dozen seconds) and finish in a function called test_passed(). There is a standard pattern to the file structure and the makefiles to build the tests automatically within the framework. The tests are run by downloading each version of each test (one test for each of the four Scalios libraries) and running them on an emulation environment. When the test hits test_passed() the next test can be loaded and run. This test build environment is amenable to automation should the debugging tools support this.
