wiki1062: TestFramework
There is a common test framework to support testing. This includes standard test code and a standard build environment.

All tests are designed to run for a short period of time (no more than a dozen seconds) and to finish in test_passed() (or otherwise end up in test_failed(), os_assertionfailure() or a hardware error trap). The tests are all designed to sit within a test directory under the builds directory and to contain three standard files:

  1. main.c: the test itself
  2. main.oil: the OIL that will build Scalios for the test
  3. build.mf: the makefile to construct a single test.

All build.mf makefiles follow the same pattern: a definition of where Scalios is installed, for what target the tests are being built, and the build options where the defaults are not appropriate. The build.mf makefiles use standard framework makefile fragments to control the building of the test executables, each in a separate subdirectory (called Extended, Standard, ExtendedNS and StandardNS for the four Scalios library builds). A 'super' makefile exists at the top level (builds) to build the entire test suite. The make files follow the standard 'all' and 'clean' targets so to build the test the command line is: 

$ make -f build.mf

from within the test subdirectory.

And to clean out tool-generated files: 

$ make -f build.mf clean

The parameters for the standard build environment are described below. A 'switch' parameter generally follows the pattern x=x to turn the switch on and x=NO_x to turn it off.

  • INSTALL: set to where the OS is installed.
  • TARGET: set to the name of the target (e.g. dsPIC)
  • INCLUDE_HANDLE_MAP_NAME: Used in osgen.c to link in an array mapping handles to textual names for the handle.
  • USERESSCHEDULER: Indicates whether RESSCHEDULER resource is included.
  • USEPRETASKHOOK: Indicates whether the pre-task hook is to be included.
  • USEPOSTTASKHOOK: Indicates whether the post-task hook is to be included.
  • USESTARTUPHOOK: Indicates whether the startup hook is to be included.
  • USESHUTDOWNHOOK: Indicates whether the shutdown hook is to be included.
  • USEERRORHOOK: Indicates whether the error hook is to be included.
  • EXPECTSERRORS: Indicates whether the test expects to generate API errors or not. If not, and the error hook is called, the hook in framework.c will cause the test to fail.
  • OPTIMIZED_QUEUEING: Indicates whether the test is configured so that the optimized task queueing code is in operation.
  • USE_FAKE_COUNTER_DRIVER: Indicates whether the test is configured to use the fake timer driver or not.
  • SINGLETON_ALARMCOUNTER: Indicates whether the test is configured to keep to singleton alarms and counters optimization or not.

To add a new parameter the makefiles must be changed in several places. See the comments in common/makeversion.mf for details.

The common test code framework (in framework.c and framework.h) provides code that can run in all tests. This framework includes the test_passed() and test_failed() functions, the error hook, the pre/post task hooks, etc., code to translate between handles and the textual names for those handles (to aid in debugging), the setting up of an expected log of test events, and functions to match against that log as the test progresses. The framework code also include a complete fake timer driver.

The test code framework also includes support for building a test in four versions (linked against Standard, Extended, StandardNS and ExtendedNS libraries), and variants of tests (where the test code is unchanged but simply compiled with different build settings). There is a target-specific part to the framework to ensure that the main test code is target-independent. The target-specific part provides support for programmatically invoking an ISR and also hooks the fake timer driver to a specific timer driver interface.

Test failure is indicated by calling test_failed() (which typically has a breakpoint placed in it by the test operator). The parameter to the function is a string indicating which test has failed and optionally other information about the failure. The operator can use this to document a run of the tests.

Test passing is indicated by calling test_passed().
