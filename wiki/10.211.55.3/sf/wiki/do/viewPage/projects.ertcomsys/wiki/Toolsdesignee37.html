wiki1075: Toolsdesign (Version 12)

Overview

The off-line configuration tool has been developed using the Java programming language supported by the Unified Modelling Language (UML) as a design notation. Javadoc style comments have been used in order to allow the generation of standard java API documentation.

Java was selected due to its high level nature, good tool and debugging support and its platform independence. Eclipse was used as the IDE and ANT based builder is included within the source distribution. Standard Java style conventions are applied to all source code.

There is extensive use of type safe collection within the source code hence the code includes use of Java 5 generics and annotations. Therefore the configuration tool should be compiled using Java 6 language compliance, although earlier JVM bytecode (e.g. 1.4.2 may be targeted if required).

Tool architecture

The off-line configuration tool has several distinct functions to perform when used within an AUTOSAR/OSEK style application tool chain. Such distinct functionality combined with the desire to support multiple existing and future standards has led to the development of a loosely coupled architecture made up of several Java packages.

The configuration tool source code packages are as follows:

  1. OSModel (com.jkenergy.rtos.config.osmodel)
  2. OILModel (com.jkenergy.rtos.config.oilmodel)
  3. Generator (com.jkenergy.rtos.config.generator)
  4. Parser (com.jkenergy.rtos.config.parser)
  5. Serializers (com.jkenergy.rtos.config.serializers)

Key configuration tool classes

The key classes within the configuration tool implementation are:

  • com.jkenergy.rtos.config.generator.OSGen: the entry point and central control of the system import and generation process.
  • com.jkenergy.rtos.config.generator.TargetCpu: the root class of a TargetModel instantiation, i.e. the container for the other primary TargetModel objects, such as TargetTasks, TargetISRs, TargetEvents, TargetResources, TargetCounters, TargetAlarms, TargetScheduleTables etc.
  • com.jkenergy.rtos.config.generator.OSAnsiCGenerator: the main abstract OSGenerator type class that contains the majority of the C generation code. A concrete generator class is typically derived from this to provide target specific code generation.
  • com.jkenergy.rtos.config.generator.DriverManager: managers the loading and containments of TargetDriver related classes which are responsible for generation of target driver specific code.
  • com.jkenergy.rtos.config.serializers.OILSerializer: the class that coordinates the loading, parsing, OILModel population and OSModel population processes. Also includes OIL exporting mechanisms.
  • com.jkenergy.rtos.config.serializers.XMLSerializer: the class the coordinates the loading, parsing and OSModel population processes. Also includes XML exporting mechanisms.
  • com.jkenergy.rtos.config.oilmodel.OILDefinition: the root class of an OILModel instantiation. i.e. container for the top level Implementation (OIL parameter meta-data) and Application (OIL data).
  • com.jkenergy.rtos.config.osmodel.Cpu: the root class of an OSModel instantiation, i.e. the container for the other primary OSModel objects, such as Tasks, ISRs, Events, Resources, Counters, Alarms, ScheduleTables, OS, COM, Messages etc.

Flow of operation within the configuration tool

The flow of operations within the configuration tool is discussed elsewhere. A flow diagram of how these packages relate to each other is shown in the architecture flow diagram doc1012.

Portability issues.

The design of the architecture minimises the amount of effort required to generate system code for different microcontrollers. A number of Generator classes are provided that allow for the output of the majority of the code. (e.g. an ANSI C Generator generates all standard non-target specific code). Target specific generation support is added by deriving from the standard generator classes and overriding methods where required. Also a Platform Information class is used to allow common target specific parameters to be specified during target specific generator instantiation, allowing much of the target specific generation to be handled by existing generation classes.

The generation infrastructure also provides underlying support for code generation via an API designed to allow for simple file output and text formatting functionality.

Device driver generation classes are dynamically loaded and instantiated when requested from the OS configuration. This allows for future support of specific devices to be handled simply by distributing the device specific class only. This dynamical loading model will also eventually be applied to the target specific generator classes allowing for seamless support for multiple targets.
