wiki1034: Priorities (Version 10)
There are two types of priority:
  1. Software priority
  2. Interrupt priority level (IPL)

The software priority is an integer, with 0 the lowest and (at least) 16 (e.g. on the dsPIC) being the highest priority (on some platforms the top priority might be 32; see below). The machine hardware priority is the priority level that the CPU operates at. Some simple CPUs have just two levels: 0 and 1, 0 being normal level and 1 being an interrupt level. Some CPUs (like the dsPIC) have more.

There are three ways to store these priorities:

  1. os_pri: an integer which unifies hardware and software priorities into a single range, where the bottom bits code for the software priority (where ISRs are assigned software priorities higher than tasks). The top bits code for the IPL, with IPL 0 being the lowest hardware priority (all tasks have this zero value; all ISRs have a value above zero). A simple numeric comparison of two os_pri values can determine which is higher, and a single os_pri value contains all the information necessary to set a task running with the right IPL (e.g. after locking a resource shared with an ISR).
  2. os_ipl: an integer which is in a target-specific format that maps on to the interrupt control registers of the CPU.
  3. os_primask: an integer, treated like a bitmask, that stores software priorities; bit 0 of the mask corresponds to software priority 1.

A predefined constant called os_kernelipl of type os_ipl indicates the IPL of the kernel. The offline tool must generate this variable as the priority of the highest priority category 2 ISR. Example:

  • os_ipl FASTROM(os_kernelipl) = 0x1234;

0x1234 is a placeholder for a target-specific format for the IPL.

All tasks base priorities and boost priorities have an IPL of 0 (i.e. a task arbitrates and starts executing with an IPL of 0, all interrupts enabled). The idle task has a software priority of 0. No other task can have a base priority of 0 (because otherwise it would never preempt the idle task).

A set of standard macros are defined to control these three types of priority:

  • IPL_HIGHER(a, b): returns true if IPL a is higher than IPL b.
  • PRI2IPL(p): takes an os_pri and returns the os_ipl.
  • SET_IPL(x): sets the CPU IPL from an os_ipl value x.
  • SET_IPL_FROM_PRI(x): a combination of SET_IPL(x) and PRI2IPL(p).
  • SET_IPL_KERNEL(): sets the CPU IPL to os_kernelipl.

Note: On the dsPIC there is a 1:1 mapping between the top bits of os_pri and os_ipl. This is just lucky the way the bits fall out into the status register. With other targets the os_ipl value can be used as an index into a lookup table to obtain the os_ipl value.

The OIL attribute PRIORITY in the TASK object (a uint32) is defined to start at 0 for the lowest priority task. These values are used only to indicate relative orderings between the tasks. The offline tool maps this ordering into software priorities with the lowest task having a priority of 1 (the idle task is assigned 0). The tool ensures that the range of software priorities is contiguous, i.e. 'squashed down' so there are no gaps, then the code generation part merges software priorities with hardware priorities to give os_ipl values.
