wiki1062: TestFramework (Version 2)
The builds/common/framework.ch files provide support functions for the test, such as test_passed() and test_failed() functions, default error hook, default pre/post task hook, etc.

The framework includes makefile building, including target-specific support. The tests all follow the same form by consisting of a test code file (main.c), a test OIL file (main.old), and a makefile (build.mf). The makefile can invoke the standard building framework (makeall.mf) with certain parameters set to indicate which test to build and how to build it. The makeall.mf makefile builds the tests in all four library forms, invoking makeversion.mf for each version.

The parameters passed to makeall.mf are described below:

  • INSTALL: set to where the OS is installed.
  • TARGET: set to the name of the target (e.g. "dsPIC")
  • INCLUDE_HANDLE_MAP_NAME: set to either "INCLUDE_HANDLE_MAP_NAME" or "NO_INCLUDE_HANDLE_MAP_NAME". Used in osgen.c to link in an array mapping handles to textual names for the handle.
  • USERESSCHEDULER: set to either "USERESSCHEDULER" or "NO_ USERESSCHEDULER". Indicates whether RESSCHEDULER resource is included.
  • USEPRETASKHOOK: set to either "USEPRETASKHOOK" or "NO_ USEPRETASKHOOK". Indicates whether the pre-task hook is to be included.
  • USEPOSTTASKHOOK: set to either "USEPOSTTASKHOOK" or "NO_ USEPOSTTASKHOOK". Indicates whether the post-task hook is to be included.
  • USESTARTUPHOOK: set to either "USESTARTUPHOOK" or "NO_USESTARTUPHOOK". Indicates whether the startup hook is to be included.
  • USESHUTDOWNHOOK: set to either "USESHUTDOWNHOOK" or "NO_USESHUTDOWNHOOK". Indicates whether the shutdown hook is to be included.
  • USEERRORHOOK: set to either "USEERRORHOOK" or "NO_USEERRORHOOK". Indicates whether the error hook is to be included.
  • EXPECTSERRORS: set to either "EXPECTSERRORS" or "NO_EXPECTSERRORS". Indicates whether the test expects to generate API errors or not. If not, and the error hook is called, the default hook in framework.c will cause the test to fail.

Test failure is indicated by calling a framework-provided "test failed" function (which typically has a breakpoint placed in it by the test operator). The parameter to the function is a string indicating which test has failed and optionally other information about the failure. The operator can use this to document a run of the tests.

Test passing is indicated by returning from the standard test entry function.
